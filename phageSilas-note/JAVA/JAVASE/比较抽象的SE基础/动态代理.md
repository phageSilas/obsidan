# 作用
**在不修改原有业务代码的前提下，为代码“悄悄地”增加新的功能（即无侵入式增强）。**

为了让你更好地理解它的用处，我们可以打个通俗的比方：**动态代理就像是明星的“经纪人”**。 明星（目标对象）只需要专注于唱歌、演戏（核心业务逻辑）；而像签合同、安排行程、应对媒体、出场安保等繁琐的事情（非核心的通用逻辑），全都由经纪人（代理对象）在明星出场前或出场后代为处理。

在实际的软件开发中，动态代理主要用来解决**代码重复**和**业务逻辑与非业务逻辑耦合**的问题。以下是它最核心的几个实际用途：

## 1. 实现 AOP（面向切面编程）

这是动态代理最广泛的用途。在系统开发中，有很多功能是很多类都需要用到的，比如：

- **日志记录：** 自动记录某个方法是谁在什么时间调用的，传入了什么参数，执行了多久。
    
- **性能监控：** 拦截方法调用，计算方法的执行耗时。
    
- **权限校验：** 在执行某个敏感方法之前，先拦截下来，检查当前用户是否有权限执行。
    

如果没有动态代理，你必须在每个业务方法的开头和结尾手动写这些代码，这会导致系统里充斥着大量重复的“垃圾代码”。有了动态代理，你可以把这些功能抽取成一个“切面”，在程序运行时自动“织入”到需要这些功能的方法前后，让业务代码保持绝对的纯粹。

## 2. 声明式事务管理

以 Spring 框架中的 `@Transactional` 注解为例。当你给一个方法加上这个注解后，你只需要在方法里写数据库的增删改查逻辑就行了，不需要自己写 `connection.setAutoCommit(false)` 或 `connection.commit()`。

- **底层魔法：** Spring 会用动态代理拦截这个方法。在方法执行前，代理对象帮你开启数据库事务；方法如果正常执行完，代理对象帮你提交事务；如果方法抛出异常，代理对象捕获到异常后帮你回滚事务。
    

## 3. 为“没有实现类”的接口提供服务 (RPC 与 ORM)

这是动态代理非常强大的一个“无中生有”的特性。

- **MyBatis Mapper：** 在用 MyBatis 操作数据库时，你通常只需要写一个 `UserMapper` 接口，里面写上方法名（比如 `getUserById`）并配上 SQL，**根本不需要写实现类**。为什么它能运行？因为 MyBatis 运行时通过动态代理生成了一个该接口的代理对象，当你调用 `getUserById` 时，代理对象会拦截调用，解析出对应的 SQL 语句并发给数据库执行，最后把结果封装返回给你。
    
- **RPC 框架（如 Dubbo, Feign）：** 当你调用远程服务器上的服务时，本地其实只有接口，没有具体代码。动态代理会拦截你的本地接口调用，把方法名、参数等信息打包，通过网络发送给远程服务器，服务器执行完再把结果传回来。对开发者而言，就像在调用本地方法一样自然。
    

## 4. 延迟加载 (Lazy Loading)

在类似 Hibernate 或 JPA 这样的 ORM 框架中，如果查出一个用户对象，该用户可能关联着上千篇博客文章。为了性能，框架通常不会立刻去查数据库把文章全拉出来。

- 框架会利用动态代理给“文章列表”生成一个代理对象。只有当你在代码里真实地调用了 `user.getArticles().size()` 时，代理对象才会拦截到这个动作，然后真正去执行查询数据库的操作。
    

## 5. 单元测试中的 Mock 对象

在写单元测试时，我们经常使用 Mockito 等框架来模拟（Mock）某些依赖的服务。这些框架底层也是通过动态代理动态生成模拟对象，拦截你的方法调用，并返回你预设的假数据，从而让测试与外部环境解耦。

**总结来说：** 动态代理让 Java 程序拥有了高度的**灵活性**和**解耦能力**。它让你能够将核心业务逻辑与周边服务（日志、事务、权限、网络通信等）彻底剥离开来，是现代 Java 框架（尤其是 Spring）赖以生存的基石。

**Java 动态代理（Dynamic Proxy）** 是 Java 语言中一个非常重要的高级特性。它允许我们在程序**运行时**动态地创建代理对象，并拦截对目标对象的方法调用，从而在不修改原有目标代码的情况下，对目标方法进行增强（例如：添加日志、权限校验、事务管理等）。

动态代理是实现 **AOP（面向切面编程）** 的核心底层技术。

# 为什么需要动态代理？

在了解动态代理之前，我们先看看**静态代理**。静态代理需要为每一个目标类手动编写一个代理类（实现相同的接口）。如果系统中有 100 个类需要添加日志功能，就需要写 100 个代理类，这会导致**代码极度膨胀**且**难以维护**。

**动态代理**解决了这个问题。它不需要手动编写代理类，而是在 JVM 运行期间，利用反射机制在内存中动态生成代理类的字节码。这意味着你可以用一段通用代码，代理成千上万个不同的类。

---

# 动态代理的两大主流实现

在 Java 生态中，动态代理主要分为两种实现方式：**JDK 动态代理** 和 **CGLIB 动态代理**。

### 1. JDK 原生动态代理

JDK 动态代理是 Java 标准库自带的功能。**它的核心前提是：目标对象必须实现至少一个接口。** 因为 JDK 生成的代理类也是通过实现相同的接口来伪装成目标对象的。

**核心 API：**

- **`java.lang.reflect.InvocationHandler` 接口**：这是代理逻辑的核心。你需要实现这个接口的 `invoke` 方法，所有对代理对象方法的调用，最终都会路由到这个 `invoke` 方法里。
    
- **`java.lang.reflect.Proxy` 类**：用于在运行时通过 `newProxyInstance` 方法动态生成代理对象。
    

**代码示例：**

Java

```
import java.lang.reflect.InvocationHandler;
import java.lang.reflect.Method;
import java.lang.reflect.Proxy;

// 1. 定义接口
interface UserService {
    void addUser(String name);
}

// 2. 目标类（真实实现）
class UserServiceImpl implements UserService {
    @Override
    public void addUser(String name) {
        System.out.println("成功添加用户: " + name);
    }
}

// 3. 实现 InvocationHandler (代理逻辑)
class LogHandler implements InvocationHandler {
    private Object target; // 持有被代理的目标对象

    public LogHandler(Object target) {
        this.target = target;
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println(">>> 方法执行前：记录日志，准备调用方法 " + method.getName());
        
        // 利用反射调用真实目标对象的方法
        Object result = method.invoke(target, args); 
        
        System.out.println("<<< 方法执行后：记录日志，调用结束");
        return result;
    }
}

// 4. 客户端使用
public class ProxyDemo {
    public static void main(String[] args) {
        // 创建目标对象
        UserService target = new UserServiceImpl();
        
        // 创建 Handler
        InvocationHandler handler = new LogHandler(target);
        
        // 动态生成代理对象
        UserService proxy = (UserService) Proxy.newProxyInstance(
            target.getClass().getClassLoader(), // 类加载器
            target.getClass().getInterfaces(),  // 目标类实现的接口
            handler                             // 代理逻辑处理器
        );
        
        // 通过代理对象调用方法
        proxy.addUser("张三"); 
    }
}
```

### 2. CGLIB 动态代理

如果你的目标类**没有实现任何接口**，JDK 动态代理就无能为力了。这时候就需要用到 **CGLIB (Code Generation Library)**。

**它的核心机制是：继承。** CGLIB 会在内存中动态生成一个目标类的**子类**，并重写其中的业务方法来实现代理。这也是为什么 Spring 框架经常引入 CGLIB 作为 AOP 的补充。

**核心特点与限制：**

- 不需要目标类实现接口。
    
- 因为是通过继承重写方法来实现拦截的，所以**无法代理声明为 `final` 的类和 `final` 的方法**（甚至包括 `private` 方法）。
    

---

## JDK 动态代理 vs CGLIB 动态代理

|**对比维度**|**JDK 动态代理**|**CGLIB 动态代理**|
|---|---|---|
|**底层机制**|基于接口实现（反射机制）|基于继承生成子类（字节码技术，如 ASM）|
|**前置要求**|目标类**必须**实现至少一个接口|无需接口，普通的 Java 类即可|
|**局限性**|只能代理实现了接口的方法|无法代理 `final` 类、`final` 方法和 `private` 方法|
|**依赖来源**|JDK 自带，无需外部依赖|需要引入第三方 CGLIB 依赖（Spring Core 已内嵌）|

---

## 典型应用场景

- **Spring AOP**：在 Spring 中，如果目标对象实现了接口，默认使用 JDK 动态代理；如果没有实现接口，则自动切换为 CGLIB 动态代理。
    
- **RPC 框架（如 Dubbo / gRPC）**：客户端只有接口没有实现类，通过动态代理生成接口的代理对象，代理对象在内部将方法调用转换为网络请求发送给服务端。
    
- **数据库连接池 / MyBatis Mapper**：MyBatis 中我们只需要写 Mapper 接口，不需要写实现类，底层也是通过动态代理为你生成了执行 SQL 的逻辑。
    
- **声明式事务管理**：Spring 的 `@Transactional` 注解，底层就是通过动态代理在方法执行前开启事务，方法正常执行完毕后提交事务，发生异常时回滚事务。

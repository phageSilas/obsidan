和java中的Stream流并非一个东西
之前我们在拆解秒杀异步下单逻辑时，使用到了 JVM 内部的 `BlockingQueue` 作为阻塞队列。这种方式在单节点架构下非常完美，但如果把秒杀微服务部署了多台机器，JVM 内存队列就无法跨机器共享，而且一旦服务器宕机，内存里的订单数据就会瞬间蒸发。

为了在轻量级架构中解决**分布式队列**和**消息持久化**的问题，Redis 在 5.0 版本祭出了一个真正的重量级武器：**Redis Stream**。

可以说，Stream 是 Redis 专门为了实现“消息队列（MQ）”而量身定制的数据结构，它几乎完美地填补了以前旧方案的各种大坑。

---

# 1. 为什么会有 Stream？（旧方案的痛点）

在 Stream 诞生之前，大家为了用 Redis 做消息队列，简直可以说是“绞尽脑汁”，但每种方案都有致命缺陷：

|**旧方案**|**原理**|**致命缺陷**|
|---|---|---|
|**JVM `BlockingQueue`**|内存阻塞队列|**无法分布式共享**，宕机数据全丢。|
|**Redis List**|`LPUSH` + `BRPOP` (阻塞弹出)|**不支持多播**（一条消息只能被一个微服务消费），没有状态记录，拿出来如果处理失败，消息就永远消失了（缺少 ACK 机制）。|
|**Redis Pub/Sub (发布订阅)**|频道广播|**完全不持久化**（Fire and Forget）。如果消费者断网了，期间发出的消息它永远收不到。之前讲 Redisson 的唤醒机制用它很好，但做业务队列绝对不行。|

**Redis Stream** 的出现，就是为了把上面这些缺点一次性全部干掉：它**支持持久化**、**支持多播**、**支持消费者宕机恢复**、并且内置了**消息确认（ACK）机制**。

---
# 2. Stream特点
1) 消息可回溯
2) 一个消息可以被多个消费者读取
3) 可以阻塞读取
4) 有消息漏读风险(如只读取最新消息的$)

# 3. Stream 的四大核心组件

Stream 的设计深度借鉴了目前最顶级的消息队列 **Kafka**。要理解它，主要抓住以下四个核心概念：

1. **Stream（消息链表）：** 这就是消息的本体，一个可以不断追加内容（Append-Only）的日志结构。每条消息都有一个 Redis 自动生成的唯一 ID（时间戳 + 序列号），并且消息内容可以是一组或多组 Key-Value 键值对。
    
2. **Consumer Group（消费者组）：** 这是 Stream 最灵魂的设计。你可以把多个微服务节点打包成一个“组”。
    
    - **组内竞争：** 同一个组内的多个消费者，相当于竞争关系。一条订单消息，要么被节点 A 抢走，要么被节点 B 抢走，绝不会被重复处理。（完美替代 `BlockingQueue` 的轮询分发）。
        
    - **组间广播：** 如果你创建了不同的组（比如“扣库存组”和“发短信组”），它们可以同时收到并处理这条消息。
        
3. **Last_delivered_id（游标）：** 每个消费者组内部都有一个“游标”，记录着这个组**当前消费到了哪条消息**。即便所有的消费者都宕机了，只要重启连上 Redis，组依然记得游标的位置，继续从下一条没处理的消息开始读取。
    
4. **Pending Entries List (PEL，未确认消息列表)：** 这是保证消息**绝对不丢**的终极防线。当一个消费者从组里读取了一条消息，但还没有发送 `ACK`（确认处理完毕）时，这条消息就会处于 Pending 状态。如果这个消费者处理到一半死掉了，其他消费者可以通过检查 PEL 找回这条消息并重新处理。

# 4. 基础语法命令

## (1). 生产者端：发送与控制长度

### 核心命令：`XADD`

**作用：** 向 Stream 追加一条或多条消息。 **完整语法：** `XADD key [NOMKSTREAM] [MAXLEN|MINID [=|~] threshold] *|ID field value [field value ...]`

**常用及可选参数解析：**

- **`*` (星号)**：最常用的参数。告诉 Redis 帮你自动生成一个全局唯一的递增 ID（格式是 `毫秒时间戳-序列号`，例如 `1640995200000-0`）。
    
- **`MAXLEN`**：**生产环境必加参数！** 限制 Stream 的最大长度。如果不加，Stream 会无限增长吃光 Redis 内存。
    
    - `XADD mystream MAXLEN 10000 * name zhangsan`：精确限制为 10000 条，超出时老消息会被强行截断（耗费少许 CPU 资源）。
        
    - `XADD mystream MAXLEN ~ 10000 * name zhangsan`：**（墙裂推荐）** 加上 `~` 表示“近似截断”。Redis 会在大概 10000 条左右的时候再进行截断，性能极高。
        
- **`NOMKSTREAM`**：默认情况下如果 Stream 不存在，`XADD` 会自动创建它。加上这个参数后，如果 Stream 不存在，则不创建并返回错误。
    

---

## (2). 组管理端：创建消费者组

### 核心命令：`XGROUP CREATE`

**作用：** 为指定的 Stream 创建一个消费者组。 **完整语法：** `XGROUP CREATE key groupname id|$ [MKSTREAM]`

**常用及可选参数解析：**

- **`id` 的选择（极度关键）：** 这个参数决定了这个组从哪里开始消费历史数据。
    
    - **`$`**：**最常用的选择。** 代表只消费这个组创建**之后**产生的新消息，忽略之前的历史消息。
        
    - **`0`**：代表从 Stream 的第一条历史消息开始，把所有堆积的消息全部消费一遍。
        
- **`MKSTREAM`**：如果指定的 Stream 还没被创建（比如生产者还没发过消息），直接执行 `XGROUP CREATE` 会报错。加上这个参数，Redis 会顺手帮你把空的 Stream 建好。
    

---

## (3). 消费者端：拉取与阻塞等待

### 核心命令：`XREADGROUP`

**作用：** 作为消费者组里的某个成员，去争抢并读取消息。 **完整语法：** `XREADGROUP GROUP group consumer [COUNT count] [BLOCK milliseconds] [NOACK] STREAMS key [key ...] ID [ID ...]`

**常用及可选参数解析：**

- **`GROUP group consumer`**：声明你是哪个组（`group`）里的哪个具体微服务节点（`consumer`，比如 `node-1`）。
    
- **`COUNT count`**：一次最多拉取几条消息。避免一次性拉取太多把本地内存撑爆。
    
- **`BLOCK milliseconds`**：**核心参数！完美替代 Java 里的 `take()`。** 如果 Stream 里没新消息，当前连接阻塞等待多少毫秒。
    
    - `BLOCK 0`：代表**一直死等**，直到有新消息来才被唤醒返回。
        
- **`STREAMS key >`**：这里的 `>` 是一个极为特殊的 ID，**专属于消费者组**。它表示“只读取那些**从来没有被组内其他消费者读取过**的新消息”。
    

---

## (4). 异常兜底机制：消息确认与追回

当消息被消费者读取后，就会进入这个消费者的 PEL（Pending Entries List，待确认列表）。这就涉及到后续的善后工作。

### A. 正常处理完毕：`XACK`

**作用：** 告诉 Redis 消息我已经处理完并存盘了，你可以把它从内存的 PEL 里删掉了。 **语法：** `XACK key group ID [ID ...]` _(只需要传 Stream 名字、组名和你要确认的消息 ID 即可。)_

### B. 查询挂起的消息：`XPENDING`

**作用：** 查看当前组内有哪些消息处于“已读取，但未 ACK”的待定状态。 **语法：** `XPENDING key group [start end count] [consumer]`

- 不加后面的参数，会返回概览信息（有多少条 Pending，哪个消费者占了多少条）。
    
- 加上起止 ID 和 `count`，可以看到具体的某条消息被卡了多久（`idle time`）。
    

### C. 认领死掉消费者的消息：`XCLAIM`

**作用：** 如果消费者 A 拿到消息后宕机了，消息永远在 PEL 里等待。消费者 B 可以用这个命令把 A 的消息“强行抢夺”过来自己处理。 **语法：** `XCLAIM key group consumer min-idle-time ID [ID ...]` **核心参数解析：**

- **`min-idle-time`**：最小空闲时间（毫秒）。只有当这条消息的闲置时间超过了这个阈值（比如超过了 10 分钟都没人 ACK），消费者 B 才能把它抢走。这是为了防止把别人正在正常处理的消息误抢了。

# 5. 总结：到底用不用它？

**Stream 的优势：**

- 作为内存级别的消息队列，它的**性能极其恐怖**，延迟在微秒级。
    
- 部署成本极低。如果你项目里已经有了 Redis，直接用 Stream 就能实现非常可靠的分布式异步秒杀，**不需要再去单独搭建笨重、复杂的 Kafka 或 RabbitMQ 集群**。
    

**Stream 的劣势：**

- 消息存储在 Redis 内存中，如果积累了几百万条没处理的消息，会把 Redis 的昂贵内存撑爆（虽然可以通过 `MAXLEN` 限制长度，但这会导致丢弃旧消息）。
    

**实战架构建议：**
对于中小型项目，或者秒杀这种对瞬时高并发要求极高、但整体队列堆积深度可控的场景，把前面我们代码里的 `BlockingQueue` 直接替换为 **Redis Stream**，是非常完美的进阶改造方案！
(不过我们一般使用中间件 RabbitMQ、RocketMQ、Kafka)

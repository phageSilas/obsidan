
在 Java 的世界里，`synchronized` 和 `ReentrantLock` 都是可重入锁。而在分布式架构下，**Redisson** 则是实现分布式可重入锁的绝对王者。

---

# 1. 什么是可重入锁？

**核心定义：** 一个线程如果已经获取了某把锁，那么它可以**再次进入**该锁保护的其他代码块，而不会被自己阻塞（不会导致死锁）。

**生活中的比喻：**

假设你有一把“万能钥匙”（锁）。你用这把钥匙打开了别墅的大门（拿到了锁），进入客厅。此时，你发现卧室的门也需要这把钥匙才能打开。

- **如果是不完全重入锁：** 你必须先退出大门，把钥匙还回去，然后再申请钥匙开卧室门。但这不可能，因为你人在客厅里出不去，这就变成了**死锁**。
    
- **如果是可重入锁：** 系统认出了你就是刚刚拿走钥匙的人，于是允许你直接拿着这把钥匙去开卧室的门，畅通无阻。
    

**典型的代码场景（为什么要重入）：**

Java

```
public synchronized void methodA() {
    System.out.println("进入方法 A");
    // methodA 已经拿到了锁，现在它要调用 methodB
    methodB(); 
}

public synchronized void methodB() {
    System.out.println("进入方法 B");
}
```

如果锁不可重入，线程在 `methodA` 里去调用 `methodB` 时，就会卡死在 `methodB` 的门外，一直苦等自己把锁释放。

# 2. 可重入锁的底层机制

要实现“可重入”，锁的底层必须记录两道核心信息：**“这把锁现在是谁的？”** 以及 **“他重入了几次？”**

它的运行机制如下：
使用redis中的hash key,即同时记录线程标识和重入(上锁)次数
![[JAVA/Redis/锁/Attachment/image-1.png]]
1. **拥有者标识 (Owner)：** 锁内部需要记录当前拿到锁的线程 ID。
    
2. **重入计数器 (Counter)：** 初始值为 0。
    
3. **加锁逻辑：**
    
    - 如果锁是空的（Counter == 0），当前线程拿到锁，把 Owner 设为自己，Counter 记为 1。
        
    - 如果锁已经被拿了，判断 Owner 是不是自己：
        
        - **如果是自己：** 直接放行，并且把 Counter 加 1（Counter = 2）。
            
        - **如果不是自己：** 乖乖排队阻塞等待。
            
4. **解锁逻辑：**
    
    - 线程执行完业务释放锁，不是直接把锁扔掉，而是把 Counter 减 1。
        
    - **当且仅当 Counter 减到 0 时**，才彻底清空 Owner 标识，把锁真正释放给其他线程。

# 3. 使用 Redisson 实现分布式可重入锁

单机环境下的可重入锁好做（因为 JVM 知道当前是哪个线程），但在**分布式环境（多台服务器、多个 JVM）**下，Redis 怎么知道来请求的线程是不是同一个呢？

Redisson 非常巧妙地利用了 Redis 的 **Hash（哈希）数据结构** 和 **Lua 脚本** 完美解决了这个问题。

### A. Redisson 的底层存储结构

在 Redis 中，Redisson 把锁存成了一个 Hash 类型，长这样：

- **Key (锁的名字):** 比如 `lock:product:1`
    
- **Field (拥有者):** `UUID + ThreadID`（比如 `90a42f6c-8...:42`）。用 UUID 区分不同机器，ThreadID 区分同一机器上的不同线程。
    
- **Value (计数器):** 记录重入的次数（1, 2, 3...）。
    

### B. 核心实现代码

Redisson 的使用非常傻瓜化，它完美实现了 **JUC** 里面 `java.util.concurrent.locks.Lock` 接口的规范。

**1. 引入依赖 & 配置 RedissonClient (Spring Boot 环境下通常有 starter 搞定)**

**2. 业务代码演示：**

Java

```
import org.redisson.api.RLock;
import org.redisson.api.RedissonClient;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;

@Service
public class OrderService {

    @Autowired
    private RedissonClient redissonClient;

    public void createOrder(Long productId) {
        // 1. 获取一个名为 "lock:product:{id}" 的锁对象
        RLock lock = redissonClient.getLock("lock:product:" + productId);

        // 2. 加锁 (这一步底层就是去 Redis 里写 Hash 和计数器)
        // lock() 具有自动阻塞机制，且自带看门狗 (Watchdog) 自动续期
        lock.lock(); 
        try {
            System.out.println("【下单逻辑】拿到锁，当前线程：" + Thread.currentThread().getId());
            
            // 3. 模拟业务中调用了另一个需要同一把锁的方法
            deductStock(productId); 
            
        } finally {
            // 4. 释放锁 (计数器减 1，减到 0 才会真正删除 Redis 里的 Key)
            lock.unlock();
            System.out.println("【下单逻辑】释放锁");
        }
    }

    public void deductStock(Long productId) {
        RLock lock = redissonClient.getLock("lock:product:" + productId);
        
        // 【核心重入点】：由于还是当前线程，这里获取锁会直接成功，Redis 里计数器 +1 变成 2
        lock.lock();
        try {
            System.out.println("   【扣减库存】重入拿到了同一把锁！执行扣减逻辑...");
        } finally {
            // 释放锁 (计数器减 1 变成 1)
            lock.unlock();
            System.out.println("   【扣减库存】释放锁");
        }
    }
}
```

### C. Redisson 底层的 Lua 脚本（它是怎么保证原子的？）

为什么 Redisson 这么牛？因为不管是判断锁是否存在、判断是不是自己的线程，还是计数器加减，它全都写在了一段 **Lua 脚本**里发给 Redis 执行。Redis 执行 Lua 脚本是单线程的，这就保证了整个判断加锁逻辑的**绝对原子性**。

加锁的伪 Lua 脚本逻辑非常易懂：

Lua

```
-- KEYS[1] = 锁的名字 (例如 lock:product:1)
-- ARGV[1] = 过期时间 (默认 30s)
-- ARGV[2] = 当前线程的 UUID+ThreadID

if (redis.call('exists', KEYS[1]) == 0) then 
    -- 锁不存在，第一次加锁
    redis.call('hset', KEYS[1], ARGV[2], 1); -- 设置 Hash: key, field, value=1
    redis.call('pexpire', KEYS[1], ARGV[1]); -- 设置过期时间
    return nil; 
end;

if (redis.call('hexists', KEYS[1], ARGV[2]) == 1) then 
    -- 锁存在，且持有者是当前线程 (重入逻辑触发！)
    redis.call('hincrby', KEYS[1], ARGV[2], 1); -- 计数器 + 1
    redis.call('pexpire', KEYS[1], ARGV[1]);    -- 刷新过期时间 (续命)
    return nil; 
end;

-- 锁被别人占了，返回锁剩余的过期时间
return redis.call('pttl', KEYS[1]);
```

# 总结

如果你在分布式架构下需要保证绝对的线程安全，同时业务逻辑又比较复杂（存在方法互相调用），**果断抛弃自己手写的 `setIfAbsent` (SETNX) 锁，直接上 Redisson。** 它不仅内置了**可重入机制（Hash + Lua）**，还帮你解决了之前聊过的**锁过期误删**问题，并且自带了**看门狗（Watchdog）自动续期**，可以说是目前 Java 圈分布式锁的终极解决方案。


# 关于Redssionn的重试机制和超时续期机制
## 1. 核心抢锁逻辑：没抢到怎么办？（PubSub + Semaphore 源码）

当调用 `lock.lock()` 时，底层最终会进入 `RedissonLock` 类的 `lockInterruptibly` 方法。你可以看到一个极其经典的 `while(true)` 自旋 + 信号量阻塞的架构。

### A. Java 端的等待与唤醒

Java

```
// 核心加锁方法 (精简版)
private void lock(long leaseTime, TimeUnit unit, boolean interruptibly) throws InterruptedException {
    long threadId = Thread.currentThread().getId();
    
    // 1. 第一次尝试获取锁 (底层发 Lua 脚本)
    // 如果拿到锁，ttl 返回 null；如果没拿到，返回锁剩余的过期时间 (ttl)
    Long ttl = tryAcquire(-1, leaseTime, unit, threadId);
    
    // 拿到锁了，直接返回，去执行业务！
    if (ttl == null) {
        return;
    }

    // ==========================================
    // 2. 没拿到锁，进入【等待唤醒】机制
    // ==========================================
    
    // 订阅这个锁对应的 Redis 频道 (PubSub)
    // 频道名类似于: redisson_lock__channel_:{锁的Key}
    RFuture<RedissonLockEntry> future = subscribe(threadId);
    commandExecutor.syncSubscription(future); // 阻塞等待订阅成功

    try {
        // 死循环尝试抢锁
        while (true) {
            // 再次尝试获取锁
            ttl = tryAcquire(-1, leaseTime, unit, threadId);
            if (ttl == null) {
                break; // 抢到了！跳出循环
            }

            // 没抢到，开始利用信号量 (Semaphore) 阻塞当前线程！
            if (ttl >= 0) {
                try {
                    // 【核心魔法】：尝试获取信号量，最多等待 ttl 的时间。
                    // 这里的 getLatch() 返回的是一个 Redisson 封装的 Semaphore 实例。
                    // 线程会在这里挂起 (睡眠)，直到别的线程释放锁并发布(Publish)消息，
                    // Redisson 客户端收到消息后，会调用 semaphore.release() 唤醒这个线程！
                    future.getNow().getLatch().tryAcquire(ttl, TimeUnit.MILLISECONDS);
                } catch (InterruptedException e) {
                    // 处理中断...
                }
            }
        }
    } finally {
        // 抢到锁或者异常退出了，取消订阅频道
        unsubscribe(future, threadId);
    }
}
```

### B. Redis 端的释放与发布 (Lua 源码)

既然有人在等待频道的信号，那释放锁的人在 `unlock()` 时必须发信号。我们来看看 `unlockInnerAsync` 中的底层 Lua 脚本：

Lua

```
-- KEYS[1] = 锁的Key, KEYS[2] = 订阅的Channel频道
-- ARGV[1] = 解锁发布的消息常量 (通常是 0), ARGV[2] = 锁的有效期, ARGV[3] = 线程ID

-- 1. 检查锁还在不在，或者拥有者是不是自己
if (redis.call('hexists', KEYS[1], ARGV[3]) == 0) then 
    return nil; -- 锁不是自己的，直接返回
end; 

-- 2. 重入计数器减 1
local counter = redis.call('hincrby', KEYS[1], ARGV[3], -1); 

-- 3. 判断是否完全释放
if (counter > 0) then 
    -- 还有重入，刷新一下过期时间
    redis.call('pexpire', KEYS[1], ARGV[2]); 
    return 0; 
else 
    -- 【核心逻辑】：重入计数器归 0，彻底释放！
    redis.call('del', KEYS[1]); -- 删除 Redis 里的 Hash 锁
    -- 发布消息到频道！！通知所有 tryAcquire 阻塞的线程醒来抢锁！
    redis.call('publish', KEYS[2], ARGV[1]); 
    return 1; 
end; 
```

## 2. WatchDog 看门狗机制源码解析

看门狗的本质是一个 **Netty 的 `TimerTask`（时间轮定时任务）**。

当你调用无参的 `lock()` 时，Redisson 会调用 `tryAcquireAsync`。如果获取锁成功，就会触发一段非常经典的递归回调逻辑来启动看门狗。

### A. Java 端：定时任务的注册与自我递归

Java

```
// 获取锁成功后，触发看门狗的方法
private void scheduleExpirationRenewal(long threadId) {
    ExpirationEntry entry = new ExpirationEntry();
    ExpirationEntry oldEntry = EXPIRATION_RENEWAL_MAP.putIfAbsent(getEntryName(), entry);
    if (oldEntry != null) {
        oldEntry.addThreadId(threadId);
    } else {
        entry.addThreadId(threadId);
        
        // 【核心启动】：触发续约逻辑
        renewExpiration();
    }
}

private void renewExpiration() {
    ExpirationEntry ee = EXPIRATION_RENEWAL_MAP.get(getEntryName());
    if (ee == null) { return; }

    // 【核心任务】：向 Netty 的时间轮中丢入一个定时任务
    // 延迟执行时间为：internalLockLeaseTime / 3 (默认 30秒 / 3 = 10秒)
    Timeout task = commandExecutor.getConnectionManager().newTimeout(new TimerTask() {
        @Override
        public void run(Timeout timeout) throws Exception {
            ExpirationEntry ent = EXPIRATION_RENEWAL_MAP.get(getEntryName());
            if (ent == null) { return; }
            Long threadId = ent.getFirstThreadId();
            if (threadId == null) { return; }

            // 1. 发送 Lua 脚本去 Redis 给锁续命
            RFuture<Boolean> future = renewExpirationAsync(threadId);
            
            future.onComplete((res, e) -> {
                if (e != null) { // 发生异常，记录日志并退出
                    log.error("Can't update lock " + getRawName() + " expiration", e);
                    return;
                }
                
                // 2. 如果续命成功（锁还是我的）
                if (res) {
                    // 【神来之笔：无限递归】：自己调用自己，安排下一个 10 秒后的任务！
                    // 只要我不 unlock (不从 MAP 中移除)，这个看门狗就会一直跑下去。
                    renewExpiration();
                }
            });
        }
    }, internalLockLeaseTime / 3, TimeUnit.MILLISECONDS);

    ee.setTimeout(task); // 保存任务引用，方便 unlock 时取消任务
}
```

### B. Redis 端：看门狗的续约指令 (Lua 源码)

上面 Java 代码中 `renewExpirationAsync(threadId)` 到底发了什么给 Redis？逻辑极其简单：

Lua

```
-- KEYS[1] = 锁的Key
-- ARGV[1] = 看门狗默认时间 (30秒的毫秒值)
-- ARGV[2] = 当前线程ID

-- 如果这把锁还在，且拥有者依然是当前线程
if (redis.call('hexists', KEYS[1], ARGV[2]) == 1) then 
    -- 直接用 pexpire 重新把过期时间拉满到 30 秒！
    redis.call('pexpire', KEYS[1], ARGV[1]); 
    return 1; -- 续约成功
end; 
return 0; -- 锁已经被别人拿了或释放了，不续约
```

## 总结

结合源码看，Redisson 的设计确实非常工业级：

1. **防 CPU 满载：** 用了 `Semaphore` 把死循环阻塞挂起。
    
2. **极速唤醒：** 用了 Redis 自身的 `Pub/Sub` 发布订阅。
    
3. **无缝续约：** 用了 Netty 高效的 `HashedWheelTimer` 驱动异步定时任务，并且通过递归回调实现了只要业务没执行完，就无限续期（续命）的效果。

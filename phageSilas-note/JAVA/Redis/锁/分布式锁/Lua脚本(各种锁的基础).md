
# 1. 什么是 Lua？为什么要和 Redis 结合？

**Lua** 本身是一个极其轻量级、用 C 语言编写的脚本语言。它的设计初衷就是为了被嵌入到其他应用程序中，提供灵活的扩展和定制功能。

**为什么要放到 Redis 里用？（核心痛点）**

在普通的 Java 业务代码中，如果我们想执行一段逻辑，比如：“先去 Redis 里查一个值，如果等于 A，我就把它修改成 B”。

- **传统方式：** Java 程序先发一条 `GET` 命令给 Redis，拿到结果后在 Java 内存里判断，然后再发一条 `SET` 命令给 Redis。这中间需要**两次网络通信**，而且在两次通信的间隙，**极有可能被其他线程插队**（也就是破坏了原子性）。
    
- **Lua 脚本方式：** 你把这段逻辑写成一段 Lua 脚本，一次性发给 Redis。Redis 会把这段脚本**当成一个整体的命令**来执行。
    

---

# 2. Lua 脚本的核心特性（高频面试题）

做笔记的话，一定要记住 Lua 脚本在 Redis 里的三大优势：

1. ==**绝对的原子性（Atomicity）：** 这是最重要的特性。Redis 执行命令是单线程的，当 Redis 开始执行一段 Lua 脚本时，会把整个脚本作为一个整体去运行，**期间不会被任何其他客户端发来的命令插队**。这就完美替代了之前我们说的多步操作带来的并发问题。==
    
2. **减少网络开销：** 将多个业务逻辑打包在一条命令中发给 Redis，减少了客户端与服务端之间的网络往返次数（Round Trip Time, RTT）。
    
3. **可复用性：** 客户端发送的脚本会缓存在 Redis 中（通过 `SCRIPT LOAD` 和 `EVALSHA` 命令），其他客户端可以直接复用，不用每次都把一大段代码传过去。
    

---

# 3. 基础语法与核心 API

在 Redis 里写 Lua 脚本，你不需要学全套的 Lua 语法，只要记住怎么接收参数、怎么调用 Redis 命令就行了。

## 核心 API

在 Lua 脚本内部，用来执行 Redis 命令的方法主要是：

- `redis.call('命令名称', '参数1', '参数2'...)`
    

## 参数传递规则

为了规范，Redis 规定把传给脚本的参数分为两类：**键名（KEYS）** 和 **普通参数（ARGV）**。在 Lua 脚本里，它们是以数组（索引从 1 开始）的形式接收的：

- `KEYS[1]`, `KEYS[2]` ... ：代表传入的 Redis Key。
    
- `ARGV[1]`, `ARGV[2]` ... ：代表传入的具体 Value 或其他参数。
    

## 客户端执行命令格式

在 Redis 命令行中执行 Lua 脚本的标准格式是 `EVAL`：

> `EVAL "lua脚本内容" key的数量 [key1 key2...] [arg1 arg2...]`

# 4.例子

前面我们在聊互斥锁的时候，提到了一个“坑”：**释放锁的时候，必须判断锁是不是自己的，如果是才能删除。**

这一步如果用 Java 代码分两步写（先 Get 判断，再 Del），就不是原子的。我们来看看用 Lua 脚本怎么写这段**“安全释放锁”**的逻辑。

**Lua 脚本代码：**

Lua

```
-- KEYS[1] 代表锁的 Key，例如 "lock:product:1"
-- ARGV[1] 代表当前线程的标识（UUID+ThreadId）

-- 第一步：获取锁的值并比较
if redis.call('get', KEYS[1]) == ARGV[1] then
    -- 第二步：如果相等（说明是自己的锁），则执行删除
    return redis.call('del', KEYS[1])
else
    -- 第三步：如果不相等，说明锁已经过期被别人抢走了，返回 0 代表删除失败
    return 0
end
```

这段脚本一旦发给 Redis，Redis 就会瞬间一气呵成地执行完 `get`、判断 和 `del`，彻底断绝了误删别人锁的可能。

# 5. 使用 Lua 脚本的注意事项

虽然 Lua 脚本很强大，但在生产环境使用时必须小心：

- **绝对不要在 Lua 脚本里写耗时的逻辑！**
    
    因为 Redis 执行 Lua 脚本是原子性、排他性的（单线程阻塞）。如果你的 Lua 脚本里写了个死循环，或者做大量复杂的数学运算耗时过长，**整个 Redis 服务器会被彻底卡死**，无法响应其他任何客户端的请求，直接导致系统大面积瘫痪。
    
    _(Redis 默认配置 `lua-time-limit` 为 5000 毫秒，超时会报错，但这 5 秒内系统也是假死状态)_。




**在某些特定的架构下（比如把 Redis 当作最终的数据源），Lua 脚本确实可以替代传统的锁；但放到整个宏观的系统架构中，乐观锁和悲观锁依然是不可替代的。**

这主要取决于**你的数据源是谁**，以及**你的业务逻辑有多复杂**。

---

## 1. 场景一：纯 Redis 内存操作（Lua 确实“秒杀”了锁）

假设你正在做一个**极高并发的秒杀系统**，为了抗住千万并发，你决定所有的库存扣减全部在 Redis 里完成（这叫“Redis 预扣减库存”）。

你写了一段 Lua 脚本：

1. 检查 Redis 里的库存 `if stock > 0`
    
2. 扣减库存 `stock = stock - 1`
    
3. 记录用户已经买过 `sadd bought_users userId`
    

因为 Lua 脚本在 Redis 里是**单线程排他执行**的，这段代码天生具备绝对的原子性。

**在这个特定的微观场景下，你确实不需要再搞什么 Redisson 分布式锁，也不需要去数据库里搞悲观锁/乐观锁。** Lua 脚本自己就充当了一个“超级悲观锁”的角色，完美解决了并发超卖问题。

---

## 2. 场景二：最终数据落盘 MySQL（必须依赖 DB 锁）

虽然秒杀时我们在 Redis 里扣了库存，但商业系统最终是要把订单和真实库存写回 **MySQL 数据库**的。

这里的问题在于：**Redis 的 Lua 脚本，管不到 MySQL 里的数据。**

- 假设你的系统里，不仅有前端用户在秒杀抢购，同时后台的仓储系统在由于进货，直接去操作了 MySQL 的库存表。
    
- 此时，Redis 里的 Lua 脚本再怎么原子，也防不住别的微服务直接去改 MySQL。
    

**所以，在落盘 MySQL 的那一刻：**

你依然需要执行类似于乐观锁的 SQL：

`UPDATE goods SET stock = stock - 1 WHERE id = 1 AND stock > 0;`

只有加上这种数据库层面的锁（乐观锁或行级悲观锁），才能保证你系统的**硬盘级数据**不会出错。

---

## 3. 场景三：复杂的跨服务业务链路（Lua 无能为力）

**这是最致命的一点：Lua 脚本绝对不能包含耗时操作。**

假设一个真实的“购买”逻辑包含以下步骤：

1. 检查库存。
    
2. 调用微信/支付宝 API 查用户的支付状态（耗时 200ms）。
    
3. 调用风控微服务，判断用户是不是黄牛（耗时 100ms）。
    
4. 生成订单，写入数据库。
    
5. 扣减库存。
    

**你能不能把这 5 步全写进 Redis 的 Lua 脚本里？**

**绝对不能！**

Redis 执行 Lua 时是**单线程阻塞**的。如果你在 Lua 脚本里去调用外部 API 或者查数据库，整个 Redis 会瞬间卡死，其他所有用户的请求都会超时报错，系统当场崩溃。

**这种时候，Lua 脚本彻底没用了，你只能退回到应用程序（Java）层面：**

- 使用 **Redisson 分布式锁（悲观锁）** 锁住这个用户的购买流程。
    
- 然后在 Java 代码里慢条斯理地去调用风控、调用支付、写数据库。
    

---

## 总结：大家的“生态位”完全不同

为了好理解，你可以这么记忆它们的定位：

| **技术手段**            | **作用范围**             | **执行速度**     | **典型使用场景**                              |
| ------------------- | -------------------- | ------------ | --------------------------------------- |
| **Lua 脚本**          | **仅限 Redis 内部**的数据流转 | **极快** (微秒级) | Redis 内部状态的原子改变（如秒杀预扣减、限流器计数）。不能有外部 IO。 |
| **分布式锁 (Redisson)** | **整个微服务集群**的业务流转     | **中等**       | 协调多个微服务去执行一套复杂的、包含外部 IO 的业务流（防重复下单）。    |
| **DB 乐观/悲观锁**       | **关系型数据库 (硬盘)**      | **较慢**       | 业务最终落盘时的最后一道数据安全防线。                     |
|                     |                      |              |                                         |

**简单来说：**

Lua 脚本是 **“内存里的微观交通警”**，只管 Redis 内部这个十字路口；

分布式锁是 **“城市高架桥的调度中心”**，协调多辆车（多台服务器）不要撞车；

DB 锁是 **“最终金库的保安”**，管你外面怎么飞，进金库动钱必须要过我这关。

这就好比有了高铁（Lua），并不代表汽车（数据库锁）和飞机（分布式锁）就没用了，它们适应的距离和场景完全不一样！

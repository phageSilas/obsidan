# 作用:保证数据一致性
#  1. 悲观锁 (Pessimistic Lock)

**核心思想：“总有刁民想害朕”。** 
它认为只要自己去拿数据，肯定会有别的线程来捣乱修改。所以，不管三七二十一，**在读取数据的时刻就直接把数据锁住**，直到自己操作完释放锁，其他人才允许碰这条数据。

**常见实现方式：**

- **Java 代码层面：** `synchronized` 关键字、`ReentrantLock`。
    
- **数据库层面：** 行锁、表锁、读锁、写锁。最典型的是在 SQL 语句后面加上 `FOR UPDATE`（例如：`SELECT * FROM goods WHERE id = 1 FOR UPDATE`）。
    

**优缺点：**

- **优点：** 极其安全，严格保证了数据的一致性，绝对不会出现脏读、幻读等各种并发问题。
    
- **缺点：** 性能较差。其他线程必须阻塞等待，如果拿锁的线程卡住了，后面的线程会排起长龙；此外，过度使用极其容易引发**死锁**。

# 2. 乐观锁 (Optimistic Lock)

**核心思想：“世界充满爱”。** 
它认为并发冲突的概率很低，所以**读取数据的时候根本不上锁**。大家都可以随便读。只有在最后**提交更新的那一刻**，它才会去检查：“我拿走数据的这段时间里，有没有别人偷偷改过这条数据？”

如果有，就重试或者报错；如果没有，就更新成功。

**常见实现方式：**

乐观锁通常不依赖数据库底层的锁机制，而是通过**代码逻辑**来实现，最主流的有两种方式：

## A. 版本号机制 (Version Number)

在数据库表里加一个 `version` 字段。

1. 线程 A 查数据，拿到 `version = 1`。
    
2. 线程 A 在内存里计算完毕，准备更新。
    
3. 执行更新 SQL 的时候，带上版本号条件： `UPDATE table SET stock = stock - 1, version = version + 1 WHERE id = 1 AND version = 1;`
    
4. 如果这段时间有人改过数据，数据库里的 `version` 变成了 2，那么线程 A 的这条 SQL 就会因为 `WHERE version = 1` 匹配不到任何记录而更新失败（影响行数为 0）。线程 A 就会知道发生冲突了，选择重试或放弃。
    

## B. CAS 算法 (Compare And Swap)

CAS 是 CPU 指令级别的原子操作，常用于内存数据（比如 Java 的 `AtomicInteger`）。 它的核心逻辑是：`CAS(V, E, N)`

- **V (Value)**: 当前内存里的真实值。
    
- **E (Expected)**: 我刚才读到的旧值（预期值）。
    
- **N (New)**: 我想要写入的新值。 只有当 `V 等于 E` 时，才说明没人改过，此时把 `V` 更新为 `N`；否则就不更新，一直循环重试（自旋）。
    

**优缺点：**

- **优点：** 性能极高！因为全程没有加真正的阻塞锁，线程不需要排队和上下文切换（也就是没有“线程挂起-唤醒”的开销）。
    
- **缺点：** * 如果并发冲突极其激烈，会导致大量请求更新失败，并在那里无限循环重试（自旋），把 CPU 资源全部榨干。
    
    - 存在经典的 **ABA 问题**（数据从 A 变成了 B，又变回了 A，CAS 会以为没人动过它，通常通过加版本号解决）。

# 比较
|**维度**|**悲观锁**|**乐观锁**|
|---|---|---|
|**防范策略**|事前预防（先锁再操作）|事后检查（先操作再核对）|
|**性能开销**|阻塞等待，线程切换开销大|无阻塞，但高冲突下 CPU 消耗大|
|**并发吞吐量**|低|高|
|**适用场景**|**写多读少**的场景（冲突概率极高）|**读多写少**的场景（冲突概率极低）|

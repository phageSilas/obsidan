# 1. 核心原理

缓存雪崩是指在**极短的时间内，大量的缓存 Key 同时失效（过期）**，或者是 **Redis 节点宕机**。此时，原本应该由缓存拦截的巨量请求会像雪崩一样，瞬间全部涌入后端数据库。

**后果：** 数据库瞬时压力过大导致卡顿、连接池耗尽，甚至直接崩溃，进而引发整个系统的连锁反应（服务雪崩）。

---

# 2. 产生原因

- **Key 集中过期：** 批量设置缓存时，给大量数据设置了相同的过期时间（TTL）。
    
- **Redis 实例故障：** Redis 服务器宕机、网络抖动或集群不可用。
    
- **内存淘汰：** Redis 内存达到上限，导致大量热点数据被强制剔除。

# 3. 解决方案

针对雪崩，我们需要从“事前预防”和“事后兜底”两个维度来处理：

## A. 预防 Key 集中过期

- **随机化过期时间：** 给 Key 的过期时间加上一个**随机扰动值**（例如 1-5 分钟的随机偏移）。
    
    > $$TTL = Base\_Time + Random\_Offset$$
    > 
    > 这样可以保证 Key 在时间轴上均匀分布，不会在同一时刻集体“撤退”。
    
- **热点数据永不过期：** 对于极高频访问的元数据，不设置物理过期时间，而是由后台线程逻辑更新。
    

## B. 预防 Redis 宕机

- **高可用架构：** 部署 **Redis Sentinel（哨兵模式）** 或 **Redis Cluster（集群模式）**。一旦主节点挂了，从节点能自动顶上，保证服务不断档。
    
- **多级缓存：** 增加**本地缓存**（如 Guava Cache 或 Caffeine）。即使分布式缓存挂了，本地缓存还能扛住一部分压力。

- - **降级限流：** 给缓存业务增加降级限流策略(微服务SpringCLoud)
    

## C. 系统自我保护

- **熔断与限流：** 如果数据库已经开始排队，直接触发熔断，给用户返回“系统繁忙”的提示（如 Hystrix 或 Sentinel），避免把数据库彻底压死。
    
- **加锁排队：** 如果缓存失效，只允许一个请求去查询数据库并回写缓存，其他请求等待（虽然会降低并发，但能保住数据库）。

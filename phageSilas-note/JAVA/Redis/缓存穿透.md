简单来说，缓存穿透就是指：**用户请求的数据在缓存中没有，在数据库中也没有。**

---

# 1. 核心原理

正常情况下，请求的流程是：

1. 客户端请求数据。
    
2. 先查 **Redis**，命中则返回。
    
3. 如果 Redis 没中，再查 **数据库 (DB)**。
    
4. 如果 DB 中有，回写 Redis 并返回；如果 DB 也没有，通常返回空。
    

**穿透发生时：** 由于数据根本不存在，Redis 永远不会有缓存，导致每一笔请求都直接打在数据库上。如果有人恶意构造大量不存在的 Key（如 `ID = -1`），数据库压力会瞬间暴增，甚至宕机。

# 2. 常见的解决方案

目前业界主要通过缓存空对象和布隆过滤器两种方案来应对：

## A. 缓存空对象 (Caching Null Objects)

当数据库查询结果为空时，我们仍然把这个“空结果”或一个特殊值存入 Redis，并设置一个较短的过期时间（例如 5 分钟）。

- **优点：** 实现简单，保护了后端数据库。
    
- **缺点：** * 如果请求的 Key 每次都不一样，Redis 会存入大量的无用 Key，浪费内存。
    
    - 数据一致性压力，如果后来数据库真的插入了这条数据，缓存层会有短期的脏数据（需配合主动更新）。
        
### 示例(line7~30)
``` java
@Override  
public Result queryShopById(Long id) {  
    //1. 从redis查询店铺缓存  
    String key = RedisConstants.CACHE_SHOP_KEY + id;  
    String shopJson = stringRedisTemplate.opsForValue().get(key);// 获取缓存数据  
    //2. 判断是否存在  
    if (StrUtil.isNotBlank(shopJson)) {// 判断缓存数据是否为空,若含空字符串" ",则if条件不成立  
        //3. 存在，直接返回  
        log.debug("从redis查询店铺信息{}", shopJson);  
        Shop shop = JSONUtil.toBean(shopJson, Shop.class);// 将json转为对象  
  
        return Result.ok(shop);  
    }  
    //不是空字符串且不是null再上个if以处理,走到这一步的只有" "和null,  
    //所以这里只要不是null,那就是" ",则if条件成立  
    if (shopJson != null) {  
        // 缓存穿透,以缓存空值解决  
        return Result.fail("店铺不存在");  
    }  
  
    //4. 不存在，查询数据库  
  
    //4.1. 不存在，查询数据库  
    Shop shop = getById(id);  
    log.debug("从数据库查询店铺信息{}", shop);  
    //5. 数据库不存在，返回错误  
    if (shop == null) {  
        // 缓存穿透,以缓存空值解决,注意" "和null 不一样  
        stringRedisTemplate.opsForValue().set(key, "", RedisConstants.CACHE_NULL_TTL, TimeUnit.MINUTES);  
        return Result.fail("店铺不存在");  
    }  
    //6. 存在，写入redis  
    stringRedisTemplate.opsForValue().set(key, JSONUtil.toJsonStr(shop), RedisConstants.CACHE_SHOP_TTL, TimeUnit.MINUTES);  
  
  
    //6. 存在，写入redis,并返回给前端  
  
    return Result.ok(shop);  
}
```

**注意:**  1) 空字符串" "和null不是一个东西
	  2) `StrUtil.isNotBlank`是用来字符串是否为非空白,
	    即不为 null
		不为空字符串：""
		不为空格、全角空格、制表符、换行符，等不可见字符
			判断类型如下:
``` java
		StrUtil.isNotBlank(null) // false
		StrUtil.isNotBlank("") // false
		StrUtil.isNotBlank(" \t\n") // false
		StrUtil.isNotBlank("abc") // true
		注意：该方法与 isNotEmpty(CharSequence) 的区别是： 该方法会校验空白字符

```

## B. 布隆过滤器 (Bloom Filter)

在缓存之前加一道“屏障”。布隆过滤器是一种空间效率极高的概率型数据结构，用来判断一个元素是否在一个集合中。

- **流程：** 1. 系统启动时，将所有合法的 Key 预热进布隆过滤器。 2. 请求进来时，先过过滤器。 3. 如果过滤器说“不存在”，直接返回错误，不查 Redis 也不查 DB。 4. 如果过滤器说“可能存在”，才继续后续流程。
    
- **优点：** 内存占用极低，能有效拦截大部分非法请求。
    
- **缺点：** * 存在**误判率**（它说存在不一定真存在，但它说不存在一定真的不存在）。
    
    - 元素删除困难。

## 方法使用建议
通常建议在接口层做好**参数合法性校验**（如 ID 必须大于 0），
配合**缓存空对象**应对简单的空查询，在面对海量请求的极端场景下才引入**布隆过滤器**。
加强用户权限校验
做好热点参数的限流(微服务SpringCloud)
